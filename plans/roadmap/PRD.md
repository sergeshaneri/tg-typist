# PRD: Telegram-бот для соционического интервью с DeepSeek

## Назначение

Бот проводит соционическое интервью в Telegram и использует DeepSeek API как нейронный слой типировщика. Первая версия должна работать с одним системным промптом и одной активной перепиской пользователя. Бот принимает сообщение, сохраняет его в PostgreSQL, собирает всю историю активной сессии, отправляет ее в DeepSeek вместе с системным промптом и возвращает ответ пользователю.

Цель MVP - надежно собрать работающий контур `Telegram -> бот -> PostgreSQL -> DeepSeek -> Telegram`, чтобы потом заменить один общий промпт на оркестратор с отдельными окнами по признакам Рейнина и функциям модели А.

## Пользователи

- Типируемый пользователь: пишет боту, отвечает на вопросы, получает уточнения и предварительные выводы.
- Администратор/автор методики: обновляет системный промпт, смотрит статус сервиса, при необходимости сбрасывает сессию или выгружает диагностическую информацию без утечки приватных данных.
- Будущий агент-разработчик: берет одну задачу из `tasks.md`, реализует ее, запускает тесты и фиксирует прогресс.

## MVP

MVP включает:

- Telegram webhook service на Railway.
- PostgreSQL на Railway для пользователей, сессий, сообщений и вызовов модели.
- DeepSeek chat API adapter.
- Один системный промпт типировщика.
- Отправку всей истории активной сессии в DeepSeek.
- Команды `/start`, `/reset`, `/privacy`, `/help`.
- Базовую защиту от спама, слишком длинных сообщений, повторной параллельной обработки одного пользователя, повторной доставки Telegram webhook updates и утечки секретов в логах.
- Идемпотентную обработку Telegram updates: повторно доставленный `update_id` не должен создавать второе пользовательское сообщение или второй вызов DeepSeek.
- Admin-процедуру удаления пользовательских данных до тестирования с внешними пользователями.
- Полностью мокируемые тесты без живых Telegram и DeepSeek ключей.

## Не цели MVP

- Не строить оркестратор признаков Рейнина в первой версии.
- Не делать веб-интерфейс администратора.
- Не делать оплату, подписки или публичный marketplace.
- Не делать ручную модерацию всех переписок.
- Не использовать внешние веб-источники как источник истины по соционике.
- Не отправлять в DeepSeek переписки других пользователей или архивные сессии без явного решения.

## Основной пользовательский сценарий

1. Пользователь открывает бота и отправляет `/start`.
2. Бот создает или находит пользователя и активную сессию.
3. Бот объясняет коротко, что это интервью, и предлагает начать.
4. Пользователь отправляет сообщение.
5. Бот сохраняет входящее сообщение.
6. Бот собирает системный промпт и всю историю активной сессии.
7. Бот вызывает DeepSeek.
8. Бот сохраняет ответ ассистента.
9. Бот отправляет ответ пользователю.
10. Пользователь продолжает интервью.
11. `/reset` закрывает активную сессию и создает новую.

## Политика истории

Первая версия отправляет DeepSeek всю историю активной сессии пользователя. Это осознанное продуктовое решение для качества типирования.

Ограничения:

- История берется только из активной сессии.
- Архивные сессии не добавляются в prompt по умолчанию.
- Если DeepSeek возвращает ошибку лимита контекста, бот не должен падать. Он должен записать событие, сообщить пользователю, что диалог стал слишком длинным, и применить явно протестированный fallback.
- Fallback MVP: отправить последние сообщения в рамках настроенного лимита и записать в `model_calls` причину усечения. Суммаризация истории остается будущей задачей.

## Функциональные требования

### Telegram

- Принимать только приватные чаты в MVP, если пользователь явно не попросит групповые чаты.
- Поддерживать текстовые сообщения.
- На нетекстовые сообщения отвечать понятным сообщением, что MVP работает с текстом.
- Использовать webhook в production.
- Иметь локальный режим polling или локальный webhook runner для разработки.
- Отправлять typing action во время обработки.
- Не терять сообщение пользователя, если DeepSeek недоступен.
- Сохранять и проверять `update_id`, чтобы retry от Telegram не дублировал обработку.
- Отвечать на webhook в пределах bounded timeout. Если обработка DeepSeek остается синхронной в MVP, это должно быть покрыто idempotency и таймаутами; переход на background queue остается допустимым hardening-шагом.

### DeepSeek

- Вынести base URL, model name, API key, temperature, timeout и max retry в настройки.
- Не хардкодить конкретную модель в бизнес-логике.
- Считать DeepSeek adapter внешней зависимостью, покрываемой контрактными тестами через mock HTTP.
- Перед реализацией сверить актуальные названия моделей с официальной документацией DeepSeek. На момент планирования предпочтительные дефолты: `deepseek-v4-flash` для дешевого/быстрого MVP или `deepseek-v4-pro` для более качественного интервью; deprecated aliases не использовать как дефолт.
- Хранить системный промпт в versioned-файле или в базе с версией; MVP может начать с файла `src/tg_typist/prompts/typist_system.md`.
- Сохранять metadata вызова: статус, latency, model, token usage если API вернул, error class без секретов.

### База данных

- Хранить пользователей Telegram.
- Хранить сессии интервью.
- Хранить все user/assistant сообщения активной и архивных сессий.
- Хранить вызовы модели и ошибки.
- Поддерживать миграции.
- Иметь repository layer, чтобы тесты не зависели от SQL деталей.
- Гарантировать на уровне PostgreSQL, что у пользователя не больше одной активной сессии.
- Хранить обработанные Telegram `update_id` или эквивалентный idempotency key с уникальным ограничением.

### Команды

- `/start`: создать пользователя и активную сессию, показать короткое вступление.
- `/help`: показать доступные команды.
- `/privacy`: объяснить, что сообщения сохраняются для интервью и отправляются в DeepSeek.
- `/reset`: закрыть текущую активную сессию и начать новую.
- `/status`: опциональная admin-only команда после добавления админского allowlist.

## Нефункциональные требования

- Асинхронная обработка запросов.
- Per-user lock или очередь, чтобы два сообщения одного пользователя не перемешали историю.
- Таймауты и retries для DeepSeek.
- Rate limit на пользователя.
- Ограничение длины одного сообщения.
- Конфигурация только через env vars и файлы без секретов.
- Structured logs без полного текста сообщений.
- Health endpoint для Railway.
- Production server слушает `0.0.0.0:$PORT`, где `PORT` приходит из Railway.
- Deterministic tests, которые проходят без интернета и живых API ключей.

## Безопасность и приватность

Минимум для MVP:

- `.env` не коммитится.
- `.env.example` содержит только имена переменных и безопасные примеры.
- Telegram webhook secret проверяется в production.
- DeepSeek API key не выводится в logs, exceptions, test snapshots.
- Database URL не выводится полностью.
- Запрещено логировать полный prompt.
- User-facing `/privacy` должен честно говорить, что сообщения сохраняются и отправляются внешнему AI API.
- Добавить команду или admin-процедуру удаления пользователя до тестирования с внешними пользователями.
- До публичного запуска определить retention policy: сколько хранятся архивные сессии и как пользователь может запросить удаление.

## Acceptance Criteria

MVP считается готовым, когда:

- Новый пользователь может пройти несколько сообщений интервью в Telegram.
- В базе сохраняются пользователь, активная сессия, входящие сообщения, ответы ассистента и metadata model calls.
- DeepSeek получает системный промпт и всю историю активной сессии.
- `/reset` создает новую активную сессию и не подмешивает старую историю в prompt.
- Ошибка DeepSeek не теряет входящее сообщение и не создает ложный assistant response.
- Повторная доставка одного Telegram update не создает дубли сообщений или model calls.
- PostgreSQL constraint не позволяет создать две активные сессии одного пользователя.
- Rate limit и message length limit покрыты тестами.
- Privacy/deletion procedure существует и покрыта тестом или smoke-проверкой.
- `python scripts/validate.py` проходит локально.
- Railway deployment documented and smoke-tested.
- GitHub CI runs deterministic checks on pull request or push.

## Будущая итерация: оркестратор

После MVP архитектура должна позволить добавить:

- отдельные prompt windows по признакам Рейнина;
- отдельные prompt windows по функциям модели А;
- профиль пользователя с гипотезами и confidence;
- оркестратор, который выбирает, какие окна обновлять на каждом сообщении;
- агрегатор выводов и финальный отчет;
- сжатие или индексирование длинной истории.

MVP не должен блокировать эту эволюцию: хранение сообщений, prompt versioning и model call metadata должны быть достаточно явными.
